# Tree Appreciation — Project Requirements

## 1. Purpose

Tree Appreciation is a place-based memory project.

Its purpose is to enable people to **notice**, **record**, and **care for** specific trees over time by creating a persistent digital presence for each tree that accumulates memories, observations, and meaning.

The project is not about trees in general.
It is about *this tree*, here, over time.

---

## 2. Core Framing

### 2.1 Tree as Anchor

A tree is treated as a **stable reference point** in the physical world.

The tree:
- Exists independently of the system
- Grounds memory in place
- Outlives individual observers

The system does not own the tree.
It **orbits** the tree.

---

### 2.2 Presence, Not Profile

Each tree has a **presence**, not a profile.

A presence:
- Accumulates records over time
- Does not claim completeness
- Does not collapse the tree into data

The presence exists to support attention and care, not extraction or optimisation.

---

### 2.3 Memory as Accretion

Memories are added over time as **rings**, analogous to tree rings.

A memory ring:
- Refers to a specific moment or period
- Is created by a particular steward
- Preserves when and by whom it was added

Rings accumulate; they are not overwritten.

---

## 3. Problem Domains (Tree Appreciation–Specific)

### Domain A: Place-Based Memory

**Problem:**  
Memories tied to physical places are fragile, personal, and easily lost.

**Requirements:**
- Memories must be explicitly tied to a specific tree
- Spatial grounding must be preserved
- Later viewers must understand *where* the memory belongs

**Out of Scope:**
- Abstract memory feeds
- Detatched storytelling without place

---

### Domain B: Stewardship and Care

**Problem:**  
Places degrade when nobody feels responsible for them.

**Requirements:**
- People who contribute memories are visible as stewards
- Stewardship is voluntary, not exclusive
- Multiple stewards may coexist

**Constraints:**
- No single steward owns the tree
- Steward actions are attributable

---

### Domain C: Temporal Depth

**Problem:**  
Most digital systems privilege the present and erase the past.

**Requirements:**
- Older memories remain accessible
- The order of memories matters
- Viewers can perceive the passage of time

**Non-Goal:**
- Optimising for “latest” or “most engaging” content

---

### Domain D: Plural Experience

**Problem:**  
Different people experience the same tree differently.

**Requirements:**
- Multiple perspectives must coexist
- Memories do not need to agree
- Emotional, factual, and poetic records are all valid

**Failure to Avoid:**
- Collapsing the tree into a single narrative

---

## 4. Task Maps

### Task Map 1: Creating a Tree Presence

**Actor:** Initial steward  
**Trigger:** Encountering a tree worth noticing

Tasks:
1. Identify a specific tree
2. Establish a persistent reference to that tree
3. Create an initial presence that others may later discover

Constraints:
- The presence must clearly refer to *this* tree
- The act is additive, not authoritative

---

### Task Map 2: Adding a Memory Ring

**Actor:** Any steward  
**Trigger:** A meaningful encounter with the tree

Tasks:
1. Observe or experience the tree
2. Create a memory ring referencing the tree
3. Associate:
   - Time
   - Steward
   - Medium (text, image, audio, etc.)
4. Attach the ring to the tree presence

Constraints:
- Rings cannot modify or delete earlier rings
- The steward’s authorship is preserved

---

### Task Map 3: Viewing a Tree Over Time

**Actor:** Any visitor  
**Trigger:** Discovering a tree presence

Tasks:
1. Recognise the tree as a physical place
2. Explore accumulated memory rings
3. Perceive change over time (seasonal, social, emotional)

Constraints:
- The interface must not flatten time
- Older memories remain legible

---

### Task Map 4: Steward Continuity

**Actor:** Returning steward  
**Trigger:** Revisiting the same tree

Tasks:
1. Re-engage with the existing presence
2. Add new memory rings
3. Acknowledge prior stewards implicitly through continuity

Constraints:
- No obligation to maintain consistency
- Change itself is a valid signal

---

### Task Map 5: Shared Care Without Coordination

**Actor:** Multiple independent stewards  
**Trigger:** Ongoing engagement with the same tree

Tasks:
1. Add memories without prior agreement
2. Respect the existing presence
3. Allow meaning to emerge through accumulation

Constraints:
- No moderation requirement
- No consensus requirement

---

## 5. Focal Use Case: A Tree Remembered Over Years

### Scenario

A tree in a public or semi-public space is encountered repeatedly by different people over several years.

Some pass it daily.
Some notice it once.
Some return after a long absence.

### What the System Must Enable

1. The tree gains a persistent presence
2. Memories accumulate across seasons and life events
3. Different stewards leave different kinds of traces
4. A later visitor can sense:
   - Continuity
   - Change
   - Care

The value is not informational completeness, but **felt persistence**.

---

## 6. Non-Goals

Tree Appreciation is **not** intended to:
- Catalogue all trees
- Provide environmental analytics
- Optimise engagement or growth
- Enforce factual accuracy
- Replace physical presence

The project privileges **attention over scale**.

---

## 7. Open Questions (Tree-Specific)

To refine further:

1. **Discovery**
   - How important is spontaneous discovery vs intentional visitation?

2. **Ephemerality**
   - Should some memory rings fade, or is permanence assumed?

3. **Steward Identity**
   - Are stewards expected to remain pseudonymous, named, or flexible?

4. **Tone**
   - Should the system lean more toward:
     - Poetic reflection
     - Natural observation
     - Social memory
     - Or intentionally allow all without signalling preference?

5. **Boundaries**
   - Are non-tree presences (stones, benches, paths) in scope later, or explicitly excluded?
